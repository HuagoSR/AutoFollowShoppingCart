# 自主跟随购物车上位机软件开发计划

更新日期：2026-07-02

## 0. 当前状态

目前已经完成：

- Android Studio 环境配置。
- OpenBot 仓库 fork 与 submodule 配置。
- OpenBot Android 工程已完成最小编译与安装验证。
- 已初步阅读 `MainActivity`、`MainFragment`、`FreeRoamFragment` 等代码。
- 已确认 OpenBot 现有工程具备人物检测、控制输出和 App 交互基础。

目前尚未完成：

- 底盘、ESP32 / Arduino Nano、电机驱动联调。
- 手机端与下位机的整车闭环验证。
- 面向购物车场景的目标初始化、目标记忆与安全状态机改造。

因此，当前上位机工作的重点应当从“环境已搭好”转向“在硬件到位前先把软件闭环想清楚并尽量提前验证”。

## 1. 开发原则

### 1.1 先复用 OpenBot，再做场景化增强

首版不重写整套视觉与控制框架，而是复用 OpenBot 的：

- 手机摄像头与推理链路
- App 基础交互框架
- `Control(left, right)` 控制输出方式
- 串口 / 蓝牙到底盘控制板的现有控制路径

本项目新增部分主要集中在：

- 目标初始化与确认
- 目标记忆与目标保持
- 目标丢失后的有限搜索 / 重识别
- 面向购物场景的安全状态机

### 1.2 首版允许引入 ReID，但必须可降级运行

首版应先验证：

- OpenBot 内置人物检测在当前手机上的帧率与稳定性
- 在多人、货架、遮挡、转角场景下的人体检测可用性
- 检测结果是否足以支撑目标锁定与跟随控制
- 控制输出是否能稳定映射到后续底盘
- 轻量 ReID 特征是否能在 Android 端稳定输出并提升多人场景目标保持

本阶段的推荐策略不是“没有 ReID 就不能跟随”，而是：

- 若 ReID 模块部署成功，则把它作为目标身份匹配的强特征之一
- 若 ReID 模块尚未跑通，则系统自动退化为 `bbox + 运动连续性 + 尺寸 + 颜色特征 + 状态机`

也就是说，ReID 要作为首版推荐增强项进入设计，但不能成为系统唯一依赖。

### 1.3 首版不做完整导航

当前项目边界仍然是：

- 不做完整 SLAM
- 不做复杂路径规划
- 不做高精地图

更合适的路线是：

基于目标记忆和有限状态机，实现“安全优先的有限重定位”，而不是完整导航。

### 1.4 支持无底盘实时仿真调试

在硬件未就绪前，上位机不应只做日志输出，而应支持实时的人肉闭环调试：由一名测试者手持手机模拟购物车视角，另一名测试者作为跟随目标，App 实时给出“前进、左转、右转、停止、原地搜索”等提示，从而提前验证目标初始化、ReID、目标保持和控制方向。

## 2. 上位机模块划分

建议按下面的逻辑拆分：

```text
ShoppingCartFollow
├─ PersonDetectorAdapter
├─ TargetInitializer
├─ TargetMemory
├─ PersonTracker
├─ ReIDFeatureExtractor
├─ FollowStateMachine
├─ ControlGenerator
├─ HumanCommandInterpreter
├─ HumanCartSimulatorUI
└─ FollowDebugLogger
```

各模块职责如下：

| 模块 | 作用 | 首版建议 |
| --- | --- | --- |
| `PersonDetectorAdapter` | 读取 OpenBot 当前人物检测结果并转换为统一数据结构 | 直接复用 OpenBot 输出，少改底层 |
| `TargetInitializer` | 负责候选目标采集、截图、确认前逻辑 | 必做 |
| `TargetMemory` | 记录目标外观、位置、最近状态 | 必做 |
| `PersonTracker` | 在多人 / 遮挡场景中维持目标连续性 | 必做，但先做轻量级版本 |
| `ReIDFeatureExtractor` | 对候选人框提取 embedding，用于目标重识别与多人场景身份保持 | 推荐尝试，失败时允许降级 |
| `FollowStateMachine` | 管理初始化、确认、重识别、跟随、丢失、搜索、停止 | 必做 |
| `ControlGenerator` | 根据目标位置输出 `Control(left, right)` | 必做 |
| `HumanCommandInterpreter` | 将 `Control(left, right)` 翻译成“向前 / 左转 / 右转 / 停止 / 原地搜索”等人类可执行提示 | 必做 |
| `HumanCartSimulatorUI` | 在无底盘模式下实时显示状态、目标框、控制提示与调试信息 | 必做 |
| `FollowDebugLogger` | 输出状态、匹配得分、控制量、FPS 等日志 | 必做 |

## 3. 两阶段目标初始化方案

### 3.1 设计动机

如果用户站在车前完成初始化，而系统一检测到人就直接开始跟随，会有两个问题：

1. 用户不知道系统锁定的是不是自己。
2. 用户为了看手机回到车旁确认时，车辆若直接起步会有安全风险。

因此，目标初始化应拆成两个问题：

1. 系统认为自己锁定了谁。
2. 用户确认系统锁定的人是不是对的。

### 3.2 推荐流程

#### IDLE

- 车辆停止
- App 显示待命
- 仅做画面预览和人物检测

#### CAPTURE_TARGET

- 用户点击“开始目标初始化”
- 用户站到车前 1-2 m，建议保持真实被跟随姿态
- 系统在 1-2 秒内采集多帧候选目标
- 选择最稳定的一帧作为确认图

建议记录：

- `bbox`
- `bbox_area`
- `bbox_aspect_ratio`
- `upper_body_color_hist`
- `lower_body_color_hist`
- `detection_confidence`

#### LOCKED_PENDING_CONFIRM

- 系统发出提示音或语音
- App 显示候选目标截图与目标框
- 用户可执行：
  - 确认跟随此人
  - 重新拍摄
  - 取消

#### CONFIRMED_ARMED

- 用户已确认目标身份
- 车辆仍保持停止
- App 提示用户回到车前视觉范围

#### REACQUIRE_TARGET

- 用户重新回到车前
- 系统根据目标记忆进行匹配
- 只有连续稳定匹配成功后才允许进入下一阶段

建议首版规则：

- 连续 5-10 帧匹配成功
- 目标框位于画面中间区域
- 目标框面积在合理范围内
- 匹配分数高于阈值

#### READY_TO_FOLLOW

- 系统通过安全检查
- 执行 3 秒倒计时
- 倒计时结束后进入 `FOLLOW`

## 4. 目标记忆设计

### 4.1 确认用记忆

初始化阶段保存本次会话基准信息：

- `confirmed_snapshot`
- `confirmed_embedding_gallery`
- `confirmed_bbox`
- `upper_body_color_hist`
- `lower_body_color_hist`
- `bbox_area`
- `bbox_aspect_ratio`
- `initial_confidence`

其中：

- 截图主要用于确认界面展示
- 默认只在本次会话内临时保存
- 不写入相册
- 不上传网络
- 若 ReID 已接入，则初始化阶段应采样 1-2 秒内的多帧 embedding，形成小型目标特征库，而不是只保存单帧特征

### 4.2 跟随中动态记忆

跟随中还需维护短时动态信息：

- `last_bbox`
- `last_center_x`
- `last_center_y`
- `last_seen_time`
- `last_bbox_area`
- `last_motion_direction`
- `tracking_confidence`
- `lost_duration`
- `short_term_embedding`
- `long_term_embedding`

### 4.3 更新原则

目标记忆不能被快速污染。建议规则：

- 只有高匹配分数且连续稳定时才更新外观特征
- 遮挡期、多人交叉期不更新外观特征
- 检测到陌生人短暂更大或更靠近时，不立刻切换身份
- ReID embedding 采用保守更新策略，优先保留初始化阶段确认过的特征库

这部分是避免“路人经过后系统彻底跟错人”的关键。

## 5. 融合式目标身份匹配：轻量跟踪 + ReID

首版允许引入 ReID，但 ReID 不作为唯一判断依据。系统应采用检测框连续性、运动预测、尺寸约束、颜色特征、ReID embedding 和检测置信度的融合评分机制。

### 5.1 匹配特征组成

| 能力 | 首版实现方式 |
| --- | --- |
| 位置连续性 | `bbox` 中心距离、IoU |
| 尺寸稳定性 | `bbox area` 变化范围 |
| 运动趋势 | 简单速度估计或方向连续性 |
| 外观相似度 | HSV 颜色直方图 |
| ReID 相似度 | `embedding` 余弦相似度 |
| 遮挡容忍 | 连续若干帧未检测到也不立刻换人 |

### 5.2 融合评分建议

```text
match_score =
  0.30 * reid_score
+ 0.25 * position_score
+ 0.15 * motion_score
+ 0.10 * size_score
+ 0.10 * color_score
+ 0.10 * detection_confidence
```

权重在首版不要求一次定准，但建议做成可调参数。

### 5.3 ReID 模块接口

```text
输入：person bbox 裁剪图
输出：embedding 向量
```

例如：

- 128 维
- 256 维
- 512 维

匹配时可采用：

```text
reid_score = max cosine_similarity(current_embedding, any target_embedding)
```

### 5.4 ReID 运行策略

不建议对每帧、每个候选人都跑 ReID。更合理的方式是事件触发式：

- 正常 `FOLLOW`：低频运行 ReID，例如每 5-10 帧一次
- 多人靠近或交叉：对候选人运行 ReID
- 目标丢失后重新出现：对候选人运行 ReID
- 初始化 / 确认阶段：运行 ReID 建立目标 embedding 库

### 5.5 三档实现路线

| 路线 | 说明 | 是否推荐 |
| --- | --- | --- |
| A | 颜色特征 + ReID 接口预留，代码结构先按 ReID 设计 | 必做 |
| B | 预训练轻量 ReID 模型转为 TFLite，在 Android 端输出 embedding | 推荐尝试 |
| C | 自采数据微调 ReID，针对超市低机位、背影、遮挡场景做增强 | 选做增强 |

首版最稳的落地方式是：

- 先完成 A
- 同时尝试 B
- 若 B 成功，则首版正式接入 ReID
- 若 B 未完成，系统仍然可以运行

当 `match_score` 低于阈值时：

- 不直接切换到别的人
- 不因单次 ReID 失败就认定新目标
- 先进入短时丢失、等待重识别或停止状态

## 6. 状态机建议

建议将目标初始化和跟随统一进一套状态机：

```text
IDLE
  ↓
CAPTURE_TARGET
  ↓
LOCKED_PENDING_CONFIRM
  ↓
CONFIRMED_ARMED
  ↓
REACQUIRE_TARGET
  ↓
READY_TO_FOLLOW
  ↓
FOLLOW
  ↓
LOST
  ↓
SEARCH
  ↓
STOP
```

异常分支：

- `CAPTURE_TARGET` 失败 -> 回到 `IDLE` 或继续采集
- `LOCKED_PENDING_CONFIRM` 重拍 -> 回到 `CAPTURE_TARGET`
- `REACQUIRE_TARGET` 超时 -> 回到 `CONFIRMED_ARMED` 或 `IDLE`
- `FOLLOW` 中目标丢失 -> `LOST`
- `LOST` 超时未恢复 -> `SEARCH`
- `SEARCH` 超时 -> `STOP`

## 7. 无底盘实时调试模式：Human Cart Simulator

### 7.1 目标

在没有真实小车时，由一名测试者手持手机模拟购物车，另一名测试者作为跟随目标，形成“视觉输入 -> 决策输出 -> 人工执行 -> 画面反馈”的近似闭环。该模式下也能验证：

- 人物检测稳定性
- 目标初始化和确认流程
- ReID 重识别是否有效
- 多人场景下的目标保持
- 目标丢失状态机
- 控制输出合理性

### 7.2 基本运行方式

- 测试者 A 手持手机，模拟购物车视角
- 测试者 B 在前方充当目标用户
- App 不向下位机发送真实运动指令
- App 在屏幕上实时显示人类可执行提示
- 手持手机者按提示移动，形成近似闭环

### 7.3 建议显示内容

第一层应是大号动作提示，而不只是 `left / right` 数字：

- `向前慢走`
- `向前并左转`
- `向前并右转`
- `停止`
- `原地左转搜索`
- `原地右转搜索`
- `目标未确认，请不要移动`
- `目标丢失，请停止`

第二层显示目标视觉信息：

- 普通行人：白色框
- 候选目标：黄色框
- 当前确认目标：绿色框
- ReID 不可信或匹配失败：红色框

第三层显示调试量：

- 当前状态：`IDLE / CAPTURE_TARGET / ... / FOLLOW / LOST / SEARCH / STOP`
- 检测到的人数
- 当前目标框
- 候选目标框
- `reid_score`
- 匹配分数
- 丢失时长
- `left / right` 控制输出
- `forward / turn`
- FPS
- 推理耗时

### 7.4 人类指令翻译

`left / right` 更适合底盘，不适合人直接理解。建议增加解释层：

```text
Control(left, right)
  ↓
HumanCommandInterpreter
  ↓
向前 / 左转 / 右转 / 停止 / 原地搜索
```

可先采用简单规则：

```text
forward = (left + right) / 2
turn = (right - left) / 2
```

再根据阈值翻译为动作提示。

首版建议：

- 不启用主动后退提示
- 目标过近时优先提示停止，而不是让“人肉底盘”后退

### 7.5 建议日志格式

```text
timestamp,state,num_person,target_id,target_x,target_y,target_w,target_h,reid_score,match_score,forward,turn,left,right,human_command,fps
```

首版应同时支持：

- Logcat
- 屏幕叠加调试信息和大号动作提示
- 本地 CSV 日志

### 7.6 能验证与不能验证的内容

较适合提前验证：

- 目标初始化
- ReID 重识别
- 多人干扰
- 目标丢失状态机
- 控制方向是否正确
- 距离控制是否合理
- UI 是否易懂
- 端侧延迟是否可接受

不能完全替代的内容：

- 真实底盘惯性
- 电机响应
- 转向半径
- 载物稳定性
- 物理急停效果
- 通信超时保护

## 8. 控制输出建议

首版仍保持 OpenBot 兼容控制方式：

- 上位机只输出 `Control(left, right)`
- 不在 Android 端处理四驱或麦轮细节
- 由下位机负责将左右轮指令映射到实际电机

可先采用简单控制逻辑：

```text
x_error = target_center_x - image_center_x
distance_error = target_bbox_height_target - target_bbox_height

turn = K_turn * x_error
forward = K_dist * distance_error

left = forward - turn
right = forward + turn
```

补充限制：

- `left/right` 限幅
- `FOLLOW` 之外默认 `forward = 0`
- `LOST/SEARCH` 期间只允许原地小角速度扫描

## 9. 近期开发顺序建议

建议成员 A 按下面顺序推进：

1. 进一步确认 OpenBot 中人物检测结果的接入点。
2. 建立 Human Cart Simulator，先把状态、目标框、动作提示和控制量实时显示出来。
3. 实现两阶段目标初始化状态机。
4. 完成 `颜色特征 + 运动连续性 + 状态机` 的基础版本。
5. 预留 `ReIDFeatureExtractor` 接口，并尝试接入轻量 TFLite ReID 模型。
6. 实现目标 embedding 库、重识别策略和多人干扰抑制。
7. 接入目标丢失后的 `LOST / SEARCH / STOP`。
8. 硬件到位后再联调 `Control(left, right)` 到真实底盘。

---

## 10. 首版验收口径

在硬件未到位前，上位机也可以先按以下标准做阶段验收：

- 能显示人物检测框和候选目标框
- 能完成“采集 - 确认 - 重识别 - 倒计时启动”完整流程
- 能在 Human Cart Simulator 中实时显示“前进 / 左转 / 右转 / 停止 / 原地搜索”等动作提示
- 在确认前车辆控制输出保持安全停止
- 若 ReID 跑通，能在多人或重现目标场景下提升目标保持稳定性
- 若 ReID 未跑通，系统仍能退化运行，不因缺少 ReID 彻底失效
- 在多人短时干扰时不轻易切换目标
- 目标丢失后能进入 `LOST / SEARCH / STOP`
- 能输出调试日志和关键状态变化记录

---

## 11. 结论

你当前负责的上位机方向，最有价值的工作不是立刻追求复杂算法，而是先把“谁是目标、什么时候能启动、丢失后怎么安全处理”这条链路做扎实。

这份计划对应的核心创新点可以概括为：

> 确认式目标初始化 + ReID 增强目标记忆 + 人肉底盘实时仿真调试。系统先由用户确认跟随目标，再通过 ReID 与运动连续性融合评分保持目标身份；在硬件未到位时，通过手持手机模拟购物车，根据屏幕实时运动提示完成近似闭环测试，从而提前验证多人干扰、遮挡重识别和跟随控制策略。
