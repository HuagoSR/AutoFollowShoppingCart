# 自主跟随购物车上位机软件开发计划

更新日期：2026-07-02

## 0. 当前状态

目前已经完成：

- Android Studio 环境配置。
- OpenBot 仓库 fork 与 submodule 配置。
- OpenBot Android 工程已完成最小编译与安装验证。
- 已初步阅读 `MainActivity`、`MainFragment`、`FreeRoamFragment` 等代码。
- 已确认 OpenBot 现有工程具备人物检测、控制输出和 App 交互基础。

目前尚未完成：

- 底盘、ESP32 / Arduino Nano、电机驱动联调。
- 手机端与下位机的整车闭环验证。
- 面向购物车场景的目标初始化、目标记忆与安全状态机改造。

因此，当前上位机工作的重点应当从“环境已搭好”转向“在硬件到位前先把软件闭环想清楚并尽量提前验证”。

## 1. 开发原则

### 1.1 先复用 OpenBot，再做场景化增强

首版不重写整套视觉与控制框架，而是复用 OpenBot 的：

- 手机摄像头与推理链路
- App 基础交互框架
- `Control(left, right)` 控制输出方式
- 串口 / 蓝牙到底盘控制板的现有控制路径

本项目新增部分主要集中在：

- 目标初始化与确认
- 目标记忆与目标保持
- 目标丢失后的有限搜索 / 重识别
- 面向购物场景的安全状态机

### 1.2 首版不优先训练新模型

首版应先验证：

- OpenBot 内置人物检测在当前手机上的帧率与稳定性
- 在多人、货架、遮挡、转角场景下的人体检测可用性
- 检测结果是否足以支撑目标锁定与跟随控制
- 控制输出是否能稳定映射到后续底盘

只有当现有模型明显不能满足最小演示需求时，才讨论换模型或训练。

### 1.3 首版不做完整导航

当前项目边界仍然是：

- 不做完整 SLAM
- 不做复杂路径规划
- 不做高精地图

更合适的路线是：

基于目标记忆和有限状态机，实现“安全优先的有限重定位”，而不是完整导航。

### 1.4 支持无底盘调试

在硬件未就绪前，上位机应支持只做检测、显示、记录和控制量计算，不真实发动车辆。这样成员 A 可以先把主要软件逻辑推进起来。

## 2. 上位机模块划分

建议按下面的逻辑拆分：

```text
ShoppingCartFollow
├─ PersonDetectorAdapter
├─ TargetInitializer
├─ TargetMemory
├─ PersonTracker
├─ FollowStateMachine
├─ ControlGenerator
└─ FollowDebugLogger
```

各模块职责如下：

| 模块 | 作用 | 首版建议 |
| --- | --- | --- |
| `PersonDetectorAdapter` | 读取 OpenBot 当前人物检测结果并转换为统一数据结构 | 直接复用 OpenBot 输出，少改底层 |
| `TargetInitializer` | 负责候选目标采集、截图、确认前逻辑 | 必做 |
| `TargetMemory` | 记录目标外观、位置、最近状态 | 必做 |
| `PersonTracker` | 在多人 / 遮挡场景中维持目标连续性 | 必做，但先做轻量级版本 |
| `FollowStateMachine` | 管理初始化、确认、重识别、跟随、丢失、搜索、停止 | 必做 |
| `ControlGenerator` | 根据目标位置输出 `Control(left, right)` | 必做 |
| `FollowDebugLogger` | 输出状态、匹配得分、控制量、FPS 等日志 | 必做 |

## 3. 两阶段目标初始化方案

### 3.1 设计动机

如果用户站在车前完成初始化，而系统一检测到人就直接开始跟随，会有两个问题：

1. 用户不知道系统锁定的是不是自己。
2. 用户为了看手机回到车旁确认时，车辆若直接起步会有安全风险。

因此，目标初始化应拆成两个问题：

1. 系统认为自己锁定了谁。
2. 用户确认系统锁定的人是不是对的。

### 3.2 推荐流程

#### IDLE

- 车辆停止
- App 显示待命
- 仅做画面预览和人物检测

#### CAPTURE_TARGET

- 用户点击“开始目标初始化”
- 用户站到车前 1-2 m，建议保持真实被跟随姿态
- 系统在 1-2 秒内采集多帧候选目标
- 选择最稳定的一帧作为确认图

建议记录：

- `bbox`
- `bbox_area`
- `bbox_aspect_ratio`
- `upper_body_color_hist`
- `lower_body_color_hist`
- `detection_confidence`

#### LOCKED_PENDING_CONFIRM

- 系统发出提示音或语音
- App 显示候选目标截图与目标框
- 用户可执行：
  - 确认跟随此人
  - 重新拍摄
  - 取消

#### CONFIRMED_ARMED

- 用户已确认目标身份
- 车辆仍保持停止
- App 提示用户回到车前视觉范围

#### REACQUIRE_TARGET

- 用户重新回到车前
- 系统根据目标记忆进行匹配
- 只有连续稳定匹配成功后才允许进入下一阶段

建议首版规则：

- 连续 5-10 帧匹配成功
- 目标框位于画面中间区域
- 目标框面积在合理范围内
- 匹配分数高于阈值

#### READY_TO_FOLLOW

- 系统通过安全检查
- 执行 3 秒倒计时
- 倒计时结束后进入 `FOLLOW`

## 4. 目标记忆设计

### 4.1 确认用记忆

初始化阶段保存本次会话基准信息：

- `confirmed_snapshot`
- `confirmed_bbox`
- `upper_body_color_hist`
- `lower_body_color_hist`
- `bbox_area`
- `bbox_aspect_ratio`
- `initial_confidence`

其中：

- 截图主要用于确认界面展示
- 默认只在本次会话内临时保存
- 不写入相册
- 不上传网络

### 4.2 跟随中动态记忆

跟随中还需维护短时动态信息：

- `last_bbox`
- `last_center_x`
- `last_center_y`
- `last_seen_time`
- `last_bbox_area`
- `last_motion_direction`
- `tracking_confidence`
- `lost_duration`

### 4.3 更新原则

目标记忆不能被快速污染。建议规则：

- 只有高匹配分数且连续稳定时才更新外观特征
- 遮挡期、多人交叉期不更新外观特征
- 检测到陌生人短暂更大或更靠近时，不立刻切换身份

这部分是避免“路人经过后系统彻底跟错人”的关键。

## 5. 轻量级目标匹配思路

首版不引入重型 ReID，优先使用可快速实现的方法：

| 能力 | 首版实现方式 |
| --- | --- |
| 位置连续性 | `bbox` 中心距离、IoU |
| 尺寸稳定性 | `bbox area` 变化范围 |
| 运动趋势 | 简单速度估计或方向连续性 |
| 外观相似度 | HSV 颜色直方图 |
| 遮挡容忍 | 连续若干帧未检测到也不立刻换人 |

可采用简单匹配分数：

```text
match_score =
  position_score +
  size_score +
  motion_score +
  appearance_score +
  confidence_score
```

当 `match_score` 低于阈值时：

- 不直接切换到别的人
- 先进入短时丢失或等待状态

## 6. 状态机建议

建议将目标初始化和跟随统一进一套状态机：

```text
IDLE
  ↓
CAPTURE_TARGET
  ↓
LOCKED_PENDING_CONFIRM
  ↓
CONFIRMED_ARMED
  ↓
REACQUIRE_TARGET
  ↓
READY_TO_FOLLOW
  ↓
FOLLOW
  ↓
LOST
  ↓
SEARCH
  ↓
STOP
```

异常分支：

- `CAPTURE_TARGET` 失败 -> 回到 `IDLE` 或继续采集
- `LOCKED_PENDING_CONFIRM` 重拍 -> 回到 `CAPTURE_TARGET`
- `REACQUIRE_TARGET` 超时 -> 回到 `CONFIRMED_ARMED` 或 `IDLE`
- `FOLLOW` 中目标丢失 -> `LOST`
- `LOST` 超时未恢复 -> `SEARCH`
- `SEARCH` 超时 -> `STOP`

## 7. 无底盘调试模式

### 7.1 目标

在没有真实小车时，也能验证：

- 人物检测稳定性
- 目标初始化和确认流程
- 多人场景下的目标保持
- 目标丢失状态机
- 控制输出合理性

### 7.2 建议显示内容

- 当前状态：`IDLE / CAPTURE_TARGET / ... / FOLLOW / LOST / SEARCH / STOP`
- 检测到的人数
- 当前目标框
- 候选目标框
- 匹配分数
- 丢失时长
- `left / right` 控制输出
- FPS
- 推理耗时

### 7.3 建议日志格式

```text
timestamp,state,num_person,target_id,target_x,target_y,target_w,target_h,match_score,left,right,fps
```

首版可以先写入：

- Logcat
- 屏幕叠加调试信息
- 本地 CSV 日志

## 8. 控制输出建议

首版仍保持 OpenBot 兼容控制方式：

- 上位机只输出 `Control(left, right)`
- 不在 Android 端处理四驱或麦轮细节
- 由下位机负责将左右轮指令映射到实际电机

可先采用简单控制逻辑：

```text
x_error = target_center_x - image_center_x
distance_error = target_bbox_height_target - target_bbox_height

turn = K_turn * x_error
forward = K_dist * distance_error

left = forward - turn
right = forward + turn
```

补充限制：

- `left/right` 限幅
- `FOLLOW` 之外默认 `forward = 0`
- `LOST/SEARCH` 期间只允许原地小角速度扫描

## 9. 近期开发顺序建议

建议成员 A 按下面顺序推进：

1. 进一步确认 OpenBot 中人物检测结果的接入点。
2. 建立无底盘调试模式，先把检测结果、状态和控制量打出来。
3. 实现两阶段目标初始化状态机。
4. 实现轻量级目标记忆和多人场景目标保持。
5. 接入目标丢失后的 `LOST / SEARCH / STOP`。
6. 硬件到位后再联调 `Control(left, right)` 到真实底盘。

---

## 10. 首版验收口径

在硬件未到位前，上位机也可以先按以下标准做阶段验收：

- 能显示人物检测框和候选目标框
- 能完成“采集 - 确认 - 重识别 - 倒计时启动”完整流程
- 在确认前车辆控制输出保持安全停止
- 在多人短时干扰时不轻易切换目标
- 目标丢失后能进入 `LOST / SEARCH / STOP`
- 能输出调试日志和关键状态变化记录

---

## 11. 结论

你当前负责的上位机方向，最有价值的工作不是立刻追求复杂算法，而是先把“谁是目标、什么时候能启动、丢失后怎么安全处理”这条链路做扎实。

这份计划对应的核心创新点可以概括为：

> 面向真实使用场景的确认式目标初始化机制：系统不在检测到人后直接跟随，而是通过候选目标采集、声音提示、截图确认、目标重识别和倒计时启动，保证购物车跟随的是用户确认过的人，并避免初始化阶段误跟随或突然启动。
