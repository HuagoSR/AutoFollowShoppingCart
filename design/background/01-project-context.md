# 01 项目背景与定位调研

## 1. 调研目的

本文用于回答自主跟随购物车项目的三个背景问题：

1. 为什么“室内购物辅助 + 自主跟随”是一个有现实意义的原型方向。
2. 为什么首版应定位为课程可交付原型，而不是完整商业智能购物车。
3. 为什么采用 `OpenBot + Android 手机 + MCU + 差速底盘` 是合理的首版技术主线。

本文只服务于项目背景与定位，不替代后续的 OpenBot 平台调研、视觉算法调研、底盘电控调研和竞品调研。

## 2. 外部背景事实

### 2.1 购物场景存在载物与操作便利性需求

传统超市购物车依赖用户持续手推。对普通顾客来说，购物过程中经常需要同时完成推车、查看货架、拿取商品、扫码/比价、照看随身物品等动作；对老年人、行动能力下降者、携带儿童者或购买较多商品的用户来说，推车会进一步增加身体负担和操作复杂度。

WHO 关于老龄化与健康的事实页指出，全球人口正在快速老龄化；到 2030 年，全球约每 6 人中就有 1 人年龄达到 60 岁及以上，60 岁及以上人口将从 2020 年的 10 亿增加到 14 亿。该资料同时强调，安全、可达、便于步行的公共建筑与环境是支持健康老龄化的重要条件。这一背景说明，面向室内公共空间的辅助移动与载物设备具有现实场景价值。

对本项目而言，这并不意味着首版要做适老化商业产品，而是说明“让购物车减少用户持续手推负担”是一个可解释、可展示、能被课程评审理解的应用问题。

### 2.2 零售智能化推动购物车成为可改造载体

零售行业已经出现智能购物车、扫码购物车、自助结算购物车等方向。这类产品通常关注商品识别、支付结算、用户交互和门店运营效率。它们证明购物车不是单纯机械载具，也可以成为传感器、计算、交互与移动平台的集成对象。

但商用智能购物车通常系统复杂，涉及商品识别、支付风控、门店系统对接、云端服务和长期运维。本项目没有必要复制这些商业闭环。课程首版更适合从其中抽取一个工程上可闭环的子问题：在室内平坦环境中，让购物车低速、安全地跟随指定用户，并在异常时停止。

### 2.3 低成本机器人教育平台降低原型门槛

OpenBot 的公开资料将其定位为“Turning Smartphones into Robots”。OpenBot 官网描述其思路是使用常见智能手机作为机器人大脑，配合低成本轮式车体，让更多学习者能够接触机器人与 AI 教育。OpenBot GitHub README 进一步说明，该项目使用约 50 美元的小型电动车体作为机器人身体，Android 手机软件栈支持人物跟随和实时自主导航等机器人任务。

OpenBot 论文摘要也指出，传统机器人要么价格昂贵，要么在传感器、算力和通信能力上妥协；智能手机已经具备传感器、计算能力、通信能力和软件生态，因此可以作为低成本移动机器人的核心计算平台。论文还明确提到该系统能够支持 person following 和 real-time autonomous navigation 等任务。

这为本项目提供了关键可行性支撑：首版不需要从零构建完整机器人系统，可以复用 OpenBot 的手机端框架、通信思路、底盘控制模式和教学资料，把精力集中在购物车场景适配、安全策略和演示闭环上。

### 2.4 人机共处场景必须以安全约束为先

自主跟随购物车运行在室内公共空间，周围可能有人、货架、临时障碍物和狭窄通道。即使项目只是课程原型，也必须把速度、停车、急停、通信超时和目标丢失处理放在首版核心要求中。

ISO 13482:2014 面向个人护理机器人提出安全设计、保护措施和使用信息方面的要求，并覆盖移动服务机器人、身体辅助机器人和载人机器人等类型。虽然本项目不做产品认证，也不宣称符合该标准，但该标准的公开摘要说明，面向人机共处和改善生活质量的机器人需要识别碰撞、接触、误用等危险，并将风险降低到可接受水平。

因此，本项目首版采用低速运行、目标丢失取消前进、短时原地搜索、急停、通信异常停车和障碍停车，是符合人机共处原型基本安全逻辑的。

## 3. 本项目定位

### 3.1 一句话定位

本项目是一个基于 OpenBot 架构的室内自主跟随购物车课程原型，目标是在 4 周实践周期内完成低成本、低速、安全优先、可展示的跟随闭环。

### 3.2 项目不是商业智能购物车

首版不追求完整商用能力。以下内容只作为背景或后续扩展方向，不纳入首版验收：

- 自动结账与商品识别。
- 门店系统、会员系统、支付系统对接。
- 多车调度和云端运营平台。
- 完整 SLAM、高精地图和复杂路径规划。
- 长期真实超市部署与商业运维。
- 自动归位和跨区域导航。

这样收敛的原因不是这些方向不重要，而是它们会显著增加系统边界、测试难度和集成风险，容易挤压首版最关键的“看得见、跟得上、停得住”。

### 3.3 首版要解决的核心问题

首版应优先解决一个可演示、可测试、可复盘的工程闭环：

1. Android 手机能够运行 OpenBot 相关 App 或复用其架构，完成摄像头感知和基本交互。
2. MCU 能够接收手机端指令，稳定驱动差速底盘，并具备通信超时停车保护。
3. 系统能够检测并锁定单个目标用户，在室内平坦场地低速跟随。
4. 用户停下、目标丢失、障碍出现、急停触发或通信异常时，小车能够可靠进入安全状态。
5. 购物筐或载物平台安装后，整车仍能保持低速稳定运动和可控停车。

## 4. 项目约束

### 4.1 时间约束

项目周期为 4 周集中实践。该周期不足以支持复杂导航、深度定制 App、大规模模型训练、完整机械定制和长期场景测试并行完成。因此任务优先级应遵循：先基础链路，后自主跟随；先安全停车，后体验优化；先稳定演示，后功能扩展。

### 4.2 人员约束

当前按 3 人小组组织工作，比较合理的并行主线是：

- 上位机与算法：OpenBot App、视觉感知、目标锁定、跟随策略、参数调试。
- 下位机与电控：MCU 固件、通信、电机驱动、供电、急停和超时保护。
- 结构与测试：底盘装配、购物筐/载物平台、手机支架、障碍测试、演示记录和文档收口。

这种分工与仓库现有 WBS 一致，能够减少跨模块等待。

### 4.3 成本约束

仓库现有设计文档将首版核心平台成本控制在约 1500-3000 元区间，课程预算参考为每人不超过 2500 元。该约束要求首版优先选择低成本底盘、常见 MCU、普通电机驱动、手机摄像头和基础障碍传感器，不应默认引入激光雷达、高性能计算平台或复杂专用传感器。

OpenBot 的低成本手机主脑思路与该预算约束匹配。手机已经提供摄像头、计算、屏幕、通信和电池等能力，能够减少额外上位机硬件采购和集成成本。

### 4.4 场地约束

首版面向室内超市或模拟超市场地，典型条件包括平坦地面、货架通道、转角、少量行人、临时障碍物和有限光照变化。该场景适合做低速单人跟随演示，但不适合在首版承诺复杂路径规划或开放环境自主导航。

### 4.5 安全约束

安全约束应作为系统设计的硬边界，而不是附加功能。首版至少应满足：

- 最大速度受限，保持接近步行慢速。
- 目标丢失后立即取消线速度，不继续向前盲跟。
- 搜寻阶段只允许原地低速左右扫描，不允许继续前向位移。
- 搜寻超时后发送 `STOP`。
- 急停和通信异常停车由下位机侧参与保证。
- 障碍物触发时采用保守减速或停车策略。

## 5. 对技术路线的启发

### 5.1 为什么采用 Android 手机作为上位机

Android 手机自带摄像头、屏幕、IMU、通信能力和端侧推理能力，适合作为低成本机器人上位机。OpenBot 已经围绕手机主脑建立了 App、固件、车体和资料生态。本项目采用 Android 手机作为主脑，可以减少额外计算硬件，同时让视觉感知、状态显示和调试交互集中在同一设备上。

### 5.2 为什么采用 MCU 作为下位机

电机驱动、串口解析、PWM 输出、急停和通信超时保护需要稳定、直接、低延迟地控制硬件。Arduino Nano 或 ESP32 这类 MCU 成本低、资料多、调试门槛低，适合承担底层执行和安全保护。即使手机端程序异常或通信中断，MCU 仍应能够依靠超时逻辑停车。

### 5.3 为什么采用差速底盘

差速底盘结构简单，控制接口清晰，适合低速室内跟随。对首版而言，差速底盘足以完成直线、转弯、停下、原地搜索等动作；相比阿克曼转向、麦轮或复杂移动平台，差速方案更容易在短周期内完成调试和安全验证。

### 5.4 为什么首版不做完整导航

完整导航通常涉及建图、定位、路径规划、障碍物建模和长期场景鲁棒性。购物跟随首版真正需要的是“相对目标跟随”和“异常停车”，不是从任意位置自主导航到任意地点。将首版范围限定为单目标低速跟随，可以把主要风险集中在人物检测、目标锁定、运动控制和安全状态机上。

## 6. 阶段性结论

背景调研支持当前仓库已经确定的首版主线：自主跟随购物车有明确的购物便利性和室内辅助场景价值；OpenBot 证明了手机主脑低成本机器人的可行性；人机共处场景要求首版把安全停车、急停和目标丢失处理放在核心位置。

因此，01 阶段建议保持以下项目定位：

- 做课程可交付原型，不做商业智能购物车。
- 做单人、室内、低速、短距离跟随，不做复杂导航。
- 复用 OpenBot 思路和成熟模块，不从零重造上位机和机器人平台。
- 把安全状态机、通信超时、急停和障碍停车作为首版硬要求。
- 所有扩展功能都应以不破坏基础闭环为前提。
